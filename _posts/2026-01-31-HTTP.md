> 주어진 과제 : GET/POST에 대한 차이 정리 및 http헤더에 대해 최대한 많은 정보를 정리해 올 것.

<h3> GET </h3> : 클라이언트에서 서버로 어떠한 리소스로 부터 정보를 요청하기 위해 사용되는 메서드이다. 

<h4> 특징</h4> : GET를 통해 서버에 리소스를 요청할 때 웹 캐시가 요청을 가로채 서버로부터 리소스를 다시 다운로드 하는 대신 리소스의 복사본을 반환한다. 
GET 요청은 1. 브라우저 히스토리에 남으며, 2. 북마크 될 수 있다. 3. 길이 제한이 있고 4. 중요한 정보를 다루기엔 적절하지 않다.
GET은 데이터를 요청할 때만 사용된다.

<h3>POST </h3> : 클라이언트에서 서버로 리소스를 생성하거나 업데이트 하기 위해 데이터를 보낼 때 사용되는 메소드다. 

<h4>특징</h4> : POST 요청은 1. 캐시되지 않으며, 2. 브라우저 히스토리에 남지 않는다. 3. 북마크 되지 않고 4. 데이터 길이에 제한이 없다. 
<hr>

<h3>GET / POST의 차이점 </h3>
- 사용 목적 : GET은 서버의 리소스에서 데이터를 요청할 때, POST는 서버의 리소스를 생성하거나 업데이트 할 때 사용한다.

<hr>

<h3>HTTP?</h3> : 웹에서 데이터를 주고받는 서버 - 클라이언트 모델의 프로토콜이다. 웹 브라우저가 서버와 통신하는 규칙이다. 

- HTTP 통신 방법 : 사용자 -> 브라우저 -> (리소스 요청) 웹 서버 -> 리소스 응답으로 돌려줌 

- HTTP cookie, Set - Cookie 헤더 필드 : 헤더 필드는 쿠키라고 함. HTTP 서버에서 상태를 저장하는 데 사용될 수 있음. 대부분 상태가 없는 HTTP 프로토콜을 통한 상태 저장 세션. 하지만 쿠키에는 보안을 저하시키는 많은 결함이 있음.
Set-Cookie 헤더 필드를 사용하여 HTTP 서버는 이름과 값을 전달할 수 있고, 메타데이터를 사용자 에이전트에 연결한다. 
(https://datatracker.ietf.org/doc/html/rfc6265)



<hr>

<h3>HTTP Header?</h3> : 클라이언트와 서버가 요청 또는 응답으로 부가적인 정보를 전송할 수 있도록 해주는 역할.

- General header : 요청과 응답 모두에 적용. 바디에서 최종 전송되는 데이터와는 관련 X

- Request header : 패치될 리소스나 클라이언트 자체에 대한 자세한 정보를 포함. (사용자가 전송하는 메시지의 헤더)

- Response header : 위치 또는 서버 자체에 대한 정보(이름, 버전 등)와 같이 응답에 대한 부가적 정보를 가짐. (사용자가 받은 메시지의 헤더)

- Entity header : 컨텐츠 길이나 MiME 타입과 같이 Entity body에 대한 자세한 정보를 포함.
    
<h3> HTTP General Header </h3>

- 요청과 응답 모두에 적용. 하지만 Body에서 최종적으로 전송되는 데이터와는 관련 X
	
    - Date : 현재시간
    - Pragma : 캐시제어, HTTP/1.0에서 쓰던 것으로 HTTP/1.1에서는 Cache-Control이 쓰임
    - Cache-Control : 캐시를 제어할 때 사용함 
    - Upgrade : 프로토콜 변경시 사용함
    - Via : 중계(프록시)서버의 이름, 버전, 호스트 명
    - Connection : 네트워크 접속 유지에 대한 제어. HTTP/1.1은 keep-alive로 연결 유지하는 게 기본값
    - Transfer-Encoding : 사용자에게 entity를 안전하게 전송하기 위해 사용하는 incoding 형식을 지정함
    
    
    
<h3>HTTP Entity Header</h3>

- content의 길이나 MIME타입과 같이 Entity Body에 대한 자세한 정보를 포함하는 헤더
	
    - Content-Type : 리소스의 media type 명시 
    - Content-Length : byte단위를 가지는 개체 본문의 크기
    - Content-Language : 본문을 이해하는데 가장 적절한 언어
    - Content-Location : 반환된 데이터 개체의 실제 위치 
    	
        - ex)/index.html
    - Content-Disposition : 응답 메시지를 브라우저가 어떻게 처리할 것인가. 다운로드에 주로 사용됨
    - Content-Security-Policy :다른 외부파일을 import할 경우 차단할 리소스와 불러올 리소스 명시함
    	
        - ex)default-src https -> https로만 파일을 가져옴
        - ex)default-src 'self' -> 사용자 도메인에서만 가져옴
        - ex)default-src 'none' -> 외부파일은 가져올 수 없음
        
    - Content-Encoding : 리소스 압축 방식. 주로 Content-Type과 같이 사용됨. 
    - Location : 301, 302 상태코드일 때만 볼 수 있는 헤더. 서버의 응답이 다른 곳에 있다고 사용자에게 전달, 동시에 해당 위치 지정
    - Last-Modified : 리소스의 마지막 수정 날짜
    - Allow : 지원되는 HTTP 요청 메소드 
    	
        - ex) GET, HEAD, POST
    - Expires : 자원의 만료 일자
    - ETag : 리소스의 버전을 식별하는 고유한 문자열 검사기 (캐시 확인용으로 사용할 수 있음)
    
    - Content-Type
	
    - 컨텐츠의 타입 및 문자 인코딩 방식 (UTF - 8)등을 지정한다. 
    
    - 타입 및 서브타입으로 구성된다.
    
- Content-Language (공통 헤더)
	
    - 자연언어

- Content-Encoding (공통 헤더)
	
    - 데이터의 압축 방식
    - ex) Content-Encoding : gzip, deflate
    - 압축 해제 가능 : Content-Encoding 및 Content-Length를 토대		로 실행

- Content-Length (공통 헤더)
	
    - 전달되는 객체의 Byte 길이 or 크기 (10진수)
    - 응답 메시지 Body의 길이를 지정하거나 특정 지정된 객체의 길이를 	지정
    
- Contenr-Location (공통 헤더)
	
    - 해당 객체의 실제 위치

- Content-Disposition (응답 헤더)
	
    - 응답 Body를 브라우저가 어떻게 표시해야 할지 알림
    - inline인 경우 웹 페이지의 화면에 표시, attachment인 경우 다운	로드
    	
        - ex)Content-Disposition : inline
        - ex)Content-Disposition : attachment; filename ='filename.csv'
    
    - 다운로드 원하는 파일 attachment, filename옵션을 이용해 파일명 지정 가능
    - 파일용 서버를 구축할 때 사용
    

- Content-Security-Policy (응답 헤더)
	
    - 외부 파일 import시 차단 소스 및 불러올 소스 명시
    - Xss 공격에 대한 방어 가능 (허용한 외부 소스만 지정 가능하기 때문)
    	
        - ex)Content-Security-Poilcy : default-src http:
        	
            - https를 통해서만 파일을 가져옴
        
        - ex)Content-Security-Poilcy : defalt-src 'self'
        	
            - 사용자 도메인의 파일만 가져옴
        
        - ex)Content-Security-Poilcy : defalt-src 'none'
        	
            - 파일을 가져올 수 없음

- Location (응답 헤더)
	
    - 리소스가 redirect된 때에 이동된 주소, 새로 생성된 리소스 주소를 명시함
    - 300번대 응답이나 201 Created 응답일 때 페이지 이동 범위를 알려줌
    	
        - 새로 생성된 리소스의 경우 ```HTTP 상태코드인 201 created```가 반환됨
        - 300번대 응답의 경우 ```HTTP/1.1 302 Found Lacation: /```의 응답이 있을 시 브라우저는 / 주소로 redirect함
        

- Last-Modified (응답 헤더)
	
    - 리소스를 마지막으로 갱신한 날, 시간

- Transfer-Encoding (응답 헤더)
	
    - chunked (HTTP Body로 오는 메시지들이 덩어리의 형태로 오는 것
    - 동적으로 생성되어 Body의 길이를 모르는 경우 조금씩 전송 가능
    - 각 chunk마다 그 시작에 16진수 길이를 삽입해 chunk 길이를 알려줌
    
<h3>HTTP Request Header</h3>

- HTTP 요청에서 사용되지만 메시지의 콘텐츠와 관련이 없는 패치될 리소스나 클라이언트 자체에 대해 자세한 정보를 포함하는 헤더
	
    - HOST : 요청하려는 서버 호스트 이름과 포트 번호
    - User-agent : 클라이언트 프로그램 정보
    - Referer : 현재 페이지로 연결되는 링크가 있던 이전 웹 페이지의 주소
    - Accept : 클라이언트가 처리 가능한 MIME Type종류 나열
    - Accept-charset : 클라이언트가 지원가능한 문자열 인코딩 방식
    - Accept-language : 클라이언트가 지원가능한 언어 나열
    - Accept-encoding : 클라이언트가 해석 가능한 압축 방식 지정
    - IF-Modified-Since : 지정된 시간 이후 변경된 리소스 취득. 캐시가 만료되었을 때에만 데이터를 전송하는데 사용
    - Authorization : 인증 토큰을 서버로 보낼 때 쓰이는 헤더
    - Origin : 서버로 Post 요청을 보낼 때 요청이 발생한 주소를 나타내는 값. 서버 이름만 나타남
    	
        - 이 값으로 요청을 보낸 주소와 받는 주소가 다르면 CORS error 발생
    
    - Cookie : 쿠키 값 key-value로 표현되며 Set-Cookie 헤더와 함께 서버로부터 이전에 전송되어 저장된 HTTP Cookie 포함
    
    
<h3>HTTP Response Header</h3>

- 위치 또는 서버 자체에 대한 정보(이름, 버전)과 같이 응답에 대한 부가적인 정보를 가지고 있는 헤더
	
    - Server : 웹 서버의 종류
    - Age : max-age 내의 시간에서 얼마나 흘렀는지 초 단위로 알려주는 값
    - Referrer-policy : 서버 referrer 정책을 알려 주는 값
    - WWW-Athenticate : 사용자 인증이 필요한 자원을 요구할 시, 서버가 제공하는 인증 방식
    - Proxy-Authenticate : 요청한 서버가 프록시 서버인 경우 유저 인증을 위한 값
    - Set-Cookie : 서버 측에서 클라이언트에게 세션 쿠키 정보를 설정할 수 있음
